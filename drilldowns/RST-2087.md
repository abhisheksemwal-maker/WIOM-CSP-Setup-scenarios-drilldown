# RST-2087 — Restore task (reference)

Restore tasks are a **separate task type** from install. This file exists for reference to explain why some shared component logic branches on `taskType`.

## State machine (not redesigned this session)
Restore has its own state set: `AWAITING_DETAILS`, `RESPOND_NOW`, `WORKING`, `VERIFICATION_PENDING`, `RESOLVED`, `RECLASSIFIED`, `EXPIRED`, `DEFERRED`, `CANCELLED`, `ESCALATED`.

Note that `VERIFICATION_PENDING` is a state name collision with install — but the semantics are different (install: Vyom checking submission; restore: closing-side customer verification).

## Why it matters for the install redesign
Two label-bundle collisions surfaced during this session because the install and restore bundles share component-level keys:

| Key | Install meaning | Restore meaning | Resolution |
|---|---|---|---|
| `drilldown.location_section` | "जगह" | "स्थान" | Both bundles aligned to "जगह" |
| `drilldown.executor_section` | (action/info header) | different | Both bundles aligned |
| `executor.title` | action verb for install | noun for restore | Install side renamed to `executor.install.title` |
| `cta.call_customer` | `कनेक्शन को कॉल करें` | `ग्राहक को कॉल करें` | Aligned to `कनेक्शन` |
| `reason.verifying` | `ग्राहक की पुष्टि बाकी` | `जाँच हो रही है` | **Unresolved** — opposite semantics, needs per-module key |

See [`/data/collision-map.md`](../data/collision-map.md) for the full history + anti-pattern write-up.

## Key restore principle (not implemented this session but documented)
Restore's `reason.customer_denied` is "the most important Wiom voice moment in the whole app" — how the app handles a customer saying "no, service is fine" sets the tone for the entire relationship. Out of scope for this session but flagged as the most important copy-review target for the next pass.
